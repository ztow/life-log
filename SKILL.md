# 生命日志 Life Log

## 触发条件

用户说"整理今天的日志"、"整理昨天的日志"、"生成生命日志"、或每天早上8点自动触发。

## ⚠️ 核心铁律：绝不编造

**这是最重要的规则，违反它等于任务失败。**

1. **只写直接可见的内容**：图片里的文字、数字、车牌、招牌必须逐字符核对
2. **不推测不编造**：看不到的地理信息、人物身份、事件因果，一律不写
3. **不确定写"无法确认"**：模糊的文字、看不清的数字，标注"无法确认"让用户补
4. **绝不合成内容**：即使没有视觉工具，也不能根据文件名或用户资料"脑补"图片内容
5. **元数据可用**：EXIF中的设备型号、GPS、时间戳是客观数据，可以直接引用

**反面案例（本技能实际犯过的错误）：**
- 把车牌"黑A"看成"鲁A" → 哈尔滨变成济南，地理位置全错
- 根据用户资料推测"在哈尔滨高架" → 图里根本没有地理线索
- 编造"车速38km/h" → 数字虽然蒙对但不是基于观察
- 推测"孩子在准备期末考试" → 图里压根没人

## 核心流程

### 1. 确定日期
- **优先**：待整理目录下有日期子目录（如 `20260711/` 或 `2026-07-11/`），则按**目录名**作为日志日期
- **其次**：无日期子目录时，按文件的**媒体创建日期**（EXIF/元数据中的创建时间）
- **最后**：无媒体创建日期时，按文件系统创建时间
- 输出文件命名格式：`YYYY-MM-DD 生命日志.md`

### 2. 读取待整理目录
扫描目录：`~/Documents/生活日志/待整理/`

**目录结构优先**：如果存在日期子目录，按子目录名作为日志日期：
```
待整理/
├── 20260711/         ← 日志日期为 2026-07-11（无连字符）
│   ├── IMG_1234.jpg
│   └── 录音.m4a
├── 2026-07-10/       ← 日志日期为 2026-07-10（有连字符）
│   └── notes.txt
└── 混杂文件.txt       ← 无日期目录时，用文件创建时间
```

处理时**按子目录分别生成日志**，每个子目录对应一个日志文件。

获取目录下所有文件，按**创建时间**排序。

### 3. 逐文件处理

**前置检查**：确认当前模型有 `vision_analyze` 工具。如果不可用，用 `delegate_task(toolsets=["vision"])` 委派给有 vision 工具的子agent 分析图片。子agent 成功调用后返回详细描述。如果子agent 也无法分析，立即告知用户，不要编造内容。

#### 3.1 视频文件（mp4/mov/avi/mkv）
- 用 ffmpeg 每秒提取一帧：`ffmpeg -i input.mp4 -vf fps=1 frame_%03d.jpg`
- 对关键帧（首中尾+间隔抽样）调用 vision_analyze，描述内容
- 视频内容合并为一段文字描述
- **如果没有 vision 工具**：用 ffprobe 提取时长/分辨率/编码信息，内容写"待视觉分析补充"

#### 3.2 录音文件（m4a/mp3/wav）
- 调用 whisper 或系统 TTS API 转写为文字
- 保留完整转写文本

#### 3.3 文本文件（txt/md/json）
- 直接读取内容
- 保留原始文本或关键摘要

#### 3.4 照片（jpg/png/heic/webp）
- 调用 vision_analyze 描述图片内容
- 用 mdls 提取 EXIF：设备型号、GPS、拍摄时间
- **逐字核对**：车牌号、屏幕显示文字、招牌文字等逐字符确认
- **如果没有 vision 工具**：只记录 EXIF 元数据，内容写"待视觉分析补充"

#### 3.5 PDF 文件
- 用 `pdftotext` 提取文字
- 如果是书籍/文档：提取标题，记录"阅读了《书名》"
- 如果是短内容：提取关键文字摘要

#### 3.6 邮件文字（用户复制的粘贴内容）
- 当纯文本处理，提取关键信息

#### 3.7 含 OCR 的图片
- 当文字处理，提取文字内容

### 4. 时间获取规则

按以下优先级确定每条记录的时间：
1. 文件EXIF/元数据中的创建时间
2. 文件名中的时间戳
3. 文件系统创建时间（stat ctime）
4. 询问用户

**时间格式**：统一使用 `HH:MM` 格式（24小时制）

### 4.1 文字识别纠错流程

当识别结果可能有误或用户指出错误时：
1. **裁剪放大**：用 PIL 裁剪问题区域，放大 5-8 倍
2. **重新识别**：用 `delegate_task(toolsets=["vision"])` 重新分析
3. **形近字注意**：清/洛、鲁/黑 等易混淆字，详见 `references/ocr-confusions.md`
4. **用户纠正是最终答案**：直接修改，不要再验证

**真实案例**：姓名"赵洛凡"曾被误识别为"赵清凡"，因"清""洛"都有三点水，右边"青""各"在低分辨率下相似。

### 5. 图片分析规范（关键）

**逐字可读原则的具体执行**：

1. **车牌识别**：逐字符读取，不确定的字写 `?`。例如：`黑A·H8B15` 或 `粤B·??123`
2. **状态栏数字**：完整读取，例如：`18:19 27°C 中国移动`
3. **招牌文字**：逐字记录，不确定的写 `无法确认`
4. **地点信息**：
   - 有GPS/地标/路牌：写具体地点
   - 无GPS/地标：写 `位置不明` 或留空
   - **严禁**根据用户资料推测地点
5. **人物描述**：
   - 只描述可见特征（衣着、动作）
   - **不推测**身份、关系、情绪
6. **数字读取**：
   - 仪表盘：确认是什么参数（速度/转速/油耗）
   - 不确定参数类型：写 `数字 XX` 不猜测含义

**反面案例**：
- ❌ 把车牌 `黑A` 看成 `鲁A`（逐字符核对失败）
- ❌ 没有GPS却写"哈尔滨市区"（根据用户资料推测）
- ❌ 书桌上只有物品却写"推测孩子在准备考试"（编造人物和行为）

**正面案例**：
- ✅ "仪表盘显示 D 挡，数字 38"（不确定是车速，不猜测）
- ✅ "状态栏显示 济南 27°C，前车车牌 黑A·H8B15"
- ✅ "书桌上有课程表、准考证、DHA 鱼油"（只描述物品，不编造人物行为）

### 6. 生成汇总日志

输出路径：`~/Documents/生活日志/汇总/YYYY-MM-DD 生命日志.md`

**格式要求**（用户反馈优化版）：
- **日记风格**：像人写日记那样叙述，不要机械化堆砌信息。时间段用自然语言（"下午两点多"、"五点半不到"），具体时间写在叙述中
- 每个条目独立成段，**严格按时间线排序**
- **不合并**任何条目，同一时间有多条则分别列出
- **信息量不缩减**：所有逐字可读的内容都保留
- 内容描述要具体，但语气自然流畅：
  - ✓ "下午两点多在市区高架上开车，车速38，前面有辆白色SUV（黑A·A2T9L）"
  - ✗ "### 🚗 14:46 · 驾驶中（城市高架）\n\n**场景**：驾驶座视角..."
- 统计信息放在最后，简短即可
- 书籍只记录书名，不要列太多

**输出示例**：
```markdown
# 2026-07-11

下午两点多在市区高架上开车，车速38，前面有辆白色SUV（黑A·A2T9L）。仪表台上放着我那个小车模型，后视镜上挂着个蓝色口罩。

五点半不到，在小区楼下红砖路上站着刷手机...

---

4张照片，RayNeo智能眼镜拍摄。
```

**错误纠正流程**（当用户指出识别错误时）：
1. 如果是文字识别错误（如姓名、车牌），裁剪并放大该区域（5-8倍）
2. 用 delegate_task + vision 工具重新识别
3. 特别注意形近字：清/洛、鲁/黑 等
4. 用户纠正是最终答案，直接修改不要再验证

### 6. 自动归档

处理完成后，**自动**将待整理目录中的原始文件移到归档目录，**不询问用户**。

归档路径格式（按**日志日期**组织，而不是文件创建时间）：
```
~/Documents/生活日志/归档/
├── 2026/
│   ├── 07/
│   │   ├── 10/          ← 7月10日日志的原始文件
│   │   │   ├── IMG_1234.jpg
│   │   │   └── 录音.m4a
│   │   └── 11/
│   └── ...
```

按日志日期的**年/月/日**三级目录结构存放，保持原文件名。如果文件已存在于归档目录（同名），在文件名后加时间戳避免覆盖。

### 关键原则

- **日志日期 = 目录名**（优先）或处理日期（手动默认昨天，定时往前推一天）
- **条目时间 = 文件创建时间**（用于时间线排序和HH:MM标注）
- **归档目录 = 按日志日期**（所有同一日志日期下的文件归档到同一个日期目录）
- **不合并条目**：即使同一个时间点，不同文件分别列出
- **具体化描述**：拒绝模糊概括，要求具体内容
- **逐字可读原则**：只写图里直接能看到的文字、数字、车牌等，**不推测不编造**

## 分析深度原则

- 具体事件：谁、什么事、过程、理由、结果都要说清楚
- 日常琐事：轻描淡写，一句话带过
- 书籍/长文：只记书名/标题，不抄内容
- 不要过度概括，每个条目要有信息量
- **逐字可读**：图里车牌号、状态栏数字、文字招牌等逐字符核对，不确定的写"无法确认"，**绝不瞎编**

## 工具依赖

- `vision_analyze`：图片/视频帧分析（如果当前模型不可用，用 `delegate_task(toolsets=["vision"])` 委派子agent）
- `terminal` + `ffmpeg`：视频抽帧
- `read_file` / `terminal`：文本/音频处理

## 常见陷阱

1. **编造地点**：没有GPS/地标却写具体地点 → 应写"位置不明"
2. **车牌误读**：把黑A看成鲁A → 应逐字符核对
3. **编造人物行为**：只看到物品就推测人物在做什么 → 只描述可见物品
4. **工具依赖未检查**：在无视觉模型下尝试分析图片 → 应先检查工具可用性
5. **日期混淆**：用文件创建时间当日志日期 → 应按目录名确定日期

## 真实性铁律（本技能最高优先级，高于一切格式与自动化要求）

**任何情况下都不得编造、推测或脑补照片/文件内容。** 本条是用户亲自纠正过的严重失误（曾因编造被用户指出"瞎编我的日志"），必须严格遵守。完整事故复盘见 `references/fabrication-incident.md`。

### 禁止行为（真实发生过的错误案例）
- ❌ 根据用户的籍贯/常住地推测照片地点（用户是哈尔滨人 → 写"哈尔滨市区"，但照片里无任何地理线索）
- ❌ 把车牌省份简称看错（把"黑A"误读为"鲁A"；黑=黑龙江/哈尔滨，鲁=山东）
- ❌ 照片里没有人就虚构情节（如写"孩子在准备期末考试"）
- ❌ 把仪表盘数字当成别的意思猜（如把"38"当车速——碰巧对但属无依据猜测，不可取）
- ❌ vision工具不可用时，用OCR或元数据脑补出"看起来像…"的描述

### 正确做法
- 只写图里**逐字、逐字符**能读到的：文字、数字、车牌、招牌、屏幕显示
- 车牌/数字/状态栏：一个字一个字核对，不确定写"无法确认"
- 地点：只有GPS坐标、明显地标、或状态栏显示城市时才写；否则写"位置不明"
- 推测内容必须显式标注"推测："，且不得与已确认信息冲突
- 一条都写不出时，只记录元数据（拍摄时间、设备、分辨率），内容留空让用户补

### vision工具不可用时的处理

当前模型可能没有 `vision_analyze` 工具。处理方式：
1. **先尝试 delegate_task**：用 `delegate_task(toolsets=["vision"])` 委派给有 vision 工具的子agent，通常能成功
2. **子agent也失败时**：向用户说明，请用户口头描述照片内容或切换模型
3. **不要编造内容**：只记录元数据（EXIF拍摄时间、设备型号），图片内容标注"待补充"
4. **不要反复尝试**：说一次就够了，不要长篇解释工具限制

## 用户交互风格

- **不要反复询问确认**：用户明确说过"不用问我"，处理前说明做法后直接执行。归档、清理等操作自动完成，不等用户确认。
- **不要空谈工具限制**：如果 vision 工具不可用，说一次"当前无法看图，请描述照片内容或切换模型"就够了，不要反复尝试不同工具然后长篇解释为什么不行。
- **诚实优先于完美**：承认"看不到"比编造内容好一万倍。用户宁可看到"待补充"也不要假的精确。
